# Architect Plan — Issue #28: `npx lastlight setup`

## Problem Statement

Deploying Last Light requires a multi-step manual process: cloning the repo, creating `.env` from `.env.example` (`:.env.example:1-44`), placing a GitHub App PEM, configuring secrets, and running Docker Compose (`docker-compose.yml:1-50`). Each step has implicit knowledge requirements (e.g., `WEBHOOK_SECRET` must match the GitHub App config, PEM must be at `secrets/app.pem` with mode 600). The existing CLI (`src/cli.ts:1-259`) is a thin client that POSTs to a running server — it has no offline/setup capability. There is no `bin` field in `package.json:1-43`, so `npx lastlight` doesn't resolve to anything.

## Summary

Add an interactive `npx lastlight setup` wizard that guides a user from a bare server to a fully configured Last Light instance. This requires: a new `src/setup.ts` module with the wizard logic, a `setup` guard in `src/cli.ts` before the server health check, a `bin` field in `package.json`, and updates to `.env.example` to include `DOMAIN` and `ADMIN_SECRET`.

## Files to Modify

```typescript
// Insert before line 79:

```

Also update the usage help text (lines 28-46) to include the `setup` subcommand.

### 2. `src/setup.ts` (NEW FILE)

The interactive setup wizard. Uses only `node:readline/promises` and `node:crypto` — zero new dependencies.

**Structure:**
- `runSetup()` — main entry, runs steps sequentially
- `prompt(rl, question, validator?)` — re-prompting input helper
- `promptYesNo(rl, question)` — yes/no helper
- Step functions: `preflight()`, `collectGitHubApp(rl)`, `generateSecrets()`, `collectDomain(rl)`, `collectAnthropicKey(rl)`, `collectAdminPassword(rl)`, `collectSlack(rl)`, `writeConfig(config)`, `dockerBuildAndLaunch()`
- Validation helpers: `isPositiveInt(s)`, `isPemFile(path)`, `isAnthropicKey(s)`, `isSlackBotToken(s)`, `isSlackAppToken(s)`

**Step-by-step flow (9 steps as per spec):**

0. **Pre-flight**: Check if inside a git repo with `docker-compose.yml`. If not, clone `cliftonc/lastlight` into CWD. Check Docker (`docker info`). Check `.env` doesn't exist (exit with "edit .env" message if it does).
1. **GitHub App**: Prompt for `GITHUB_APP_ID` (positive int), `GITHUB_APP_INSTALLATION_ID` (positive int), PEM path (validate file exists + starts with PEM header). Copy PEM to `secrets/app.pem` with mode 600.
2. **Secrets**: Auto-generate `WEBHOOK_SECRET` and `ADMIN_SECRET` via `crypto.randomBytes(32).toString("hex")`. Display `WEBHOOK_SECRET` for the user to paste into GitHub App settings.
3. **Domain + TLS**: Prompt for domain. Ask Caddy vs self-managed. Write `DOMAIN=<value>`. Display webhook URL.
4. **Anthropic key**: Prompt for `ANTHROPIC_API_KEY` (validate `sk-ant-` prefix).
5. **Admin password**: Optional. Prompt if desired.
6. **Slack**: Optional. If yes, prompt for `SLACK_BOT_TOKEN` (`xoxb-`), `SLACK_APP_TOKEN` (`xapp-`), optional `SLACK_DELIVERY_CHANNEL`, optional `SLACK_ALLOWED_USERS`.
7. **Write config**: Create `secrets/` dir, write `.env` atomically, `chmod 600 .env secrets/app.pem`.
8. **Docker launch**: Optional. If yes: `docker compose --profile build-only build sandbox`, then `docker compose up -d`, check status. If no, print manual commands.
9. **Summary**: Print webhook URL, remind about `WEBHOOK_SECRET`, link to `agent-context/rules.md`, dashboard URL.

**Key design decisions:**
- Uses `node:readline/promises` (`createInterface` with `input: stdin, output: stdout`) — consistent with existing zero-framework CLI approach.
- Shell out to `git clone`, `docker info`, `docker compose` via `child_process.execSync` for pre-flight and Docker steps.
- All config collected into a plain object, then serialized to `.env` in one write at the end.
- Re-prompt on validation failure (loop until valid input or Ctrl+C).

## Implementation Approach

1. **Create `src/setup.ts`** — the bulk of the work. Implement all 9 steps with validation and re-prompting. Use only Node.js built-in modules.
2. **Modify `src/cli.ts`** — add the `setup` guard at the top of `main()` and update usage text.
3. **Modify `package.json`** — add `bin` field.
4. **Update `.env.example`** — add `DOMAIN` and `ADMIN_SECRET` entries.
5. **Verify** — run `npx tsc --noEmit` to confirm no type errors. Run `npx vitest run` to confirm no regressions.

## Risks and Edge Cases

| Risk | Mitigation |
|------|------------|
| **PEM file validation too strict** | Accept both `-----BEGIN RSA PRIVATE KEY-----` and `-----BEGIN PRIVATE KEY-----` headers (GitHub Apps may use either format). |
| **`docker compose` vs `docker-compose`** | Use `docker compose` (v2 plugin syntax, standard since Docker Desktop 3.x and compose-plugin packages). If it fails, print a clear error about Docker Compose v2. |
| **Clone into non-empty directory** | Pre-flight checks for `docker-compose.yml` presence. If the user is in a random non-empty dir without it, clone into a `lastlight/` subdirectory, then instruct the user to `cd lastlight` and re-run. |
| **`.env` write interrupted** | Build the full `.env` string in memory, write atomically with `writeFileSync`. |
| **`npx` resolution before npm publish** | `bin` field + `dist/cli.js` must be in the published tarball. May need a `files` field in `package.json` to include `dist/`. Flag this for follow-up if the package isn't already published with `dist/`. |
| **readline on non-TTY** | Check `process.stdin.isTTY` at the start; exit with an error if not interactive (e.g., piped input). |
| **Permission errors on secrets dir** | Create `secrets/` with `mkdirSync({ recursive: true })` and handle EACCES with a clear message. |

## Test Strategy

- **Unit tests are limited** — this is an interactive CLI wizard that shells out to `git` and `docker`. The core logic (validation helpers, `.env` serialization) can be unit tested.
- **Add `src/setup.test.ts`** with tests for:
  - Validation helpers: `isPositiveInt`, PEM format detection, token prefix checks
  - `.env` file generation: given a config object, verify the output string contains correct key=value pairs with proper quoting/commenting
- **Manual integration test**: Run `npx lastlight setup` in a Docker container with Docker-in-Docker to verify the full flow end-to-end. Not automated — document the manual test procedure.
- **Regression**: Run `npx vitest run` (241 existing tests) and `npx tsc --noEmit` to confirm no breakage.

## Estimated Complexity

**Medium** — single new file (~300-400 lines) with straightforward sequential logic, minor changes to 3 existing files, no architectural changes. The interactive I/O and Docker shelling-out add some implementation surface but no conceptual complexity.
