# Last Light Agent — Project Context

This is the working directory for the **Last Light** agent, a Hermes-based GitHub maintenance bot.

## GitHub-First Coordination

All work — regardless of origin (Discord, CLI, webhook, cron) — is tracked via GitHub issues.
If no issue exists for a request, create one in the appropriate managed repo before starting work.
Post progress updates (architect summary, implementation status, review verdict, PR link) as
comments on the tracking issue. Discord and CLI are input channels; GitHub is the coordination layer.

## Git Authentication

**IMPORTANT:** At the start of any task involving git operations:
1. Call the `setup_git_auth` MCP tool — this writes a fresh token to the credential file
2. Run the `configure_git` commands it returns — this sets up a credential helper that reads from the token file
3. After that, `git clone`, `git push`, `git pull` all work transparently — **NEVER put tokens in git URLs or commands**
4. If auth fails after ~1 hour, call `setup_git_auth` again to refresh the token — no need to reconfigure git
5. The credential file is automatically synced into the sandbox before each command by Hermes

**After cloning, always read the repo's own docs first:**
1. Check for `CLAUDE.md`, `AGENTS.md`, or `CONTRIBUTING.md` in the repo root
2. Read them before doing any analysis, testing, or implementation
3. These files contain project-specific commands, conventions, architecture notes,
   and testing setup that override generic assumptions. For example, a repo may use
   `pnpm` not `npm`, or have specific test helpers you should use.

## Review Guidelines

When reviewing pull requests, follow this priority order:

### Critical (must fix before merge)
- Security vulnerabilities (injection, auth bypass, secret exposure)
- Data loss risks
- Breaking API changes without migration path
- Missing error handling on external calls

### Important (should fix)
- Missing or inadequate tests for new functionality
- Performance regressions (N+1 queries, unbounded loops, large allocations)
- Incorrect or missing type annotations on public APIs
- Race conditions or concurrency issues

### Suggestions (nice to have)
- Code clarity improvements
- Better variable/function naming
- Opportunities to reduce duplication
- Documentation improvements

### Nits (optional)
- Style preferences not caught by linters
- Minor formatting

## Issue Triage Rules

1. **Bug reports**: Verify reproduction steps exist. Label `bug`. If missing info, add `needs-info` and comment asking for details.
2. **Feature requests**: Label `enhancement`. Check for duplicates.
3. **Questions**: Answer if possible, or label `question` and point to docs.
4. **Stale issues**: Issues with `needs-info` and no response for 14 days get a gentle reminder. 30 days → close with explanation.

## Labels

Ensure these labels exist on managed repos:
- `bug`, `enhancement`, `question`, `documentation`
- `good first issue`, `help wanted`
- `needs-info`, `needs-review`, `stale`
- `critical`, `breaking-change`
- Priority: `p0-critical`, `p1-high`, `p2-medium`, `p3-low`
